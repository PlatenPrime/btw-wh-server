# Дни переучета конкурента: требования для фронтенда

## Назначение

Нововведение добавляет в карточку конкурента календарь дней переучета (`recountDays`). Эти даты нужны, чтобы исключать ложные всплески продаж, возникающие из-за разовой корректировки остатков конкурента.

## Сущность и формат данных

- `recountDays` хранится в документе конкурента (`Konk`) как массив строк.
- Формат элемента: `YYYY-MM-DD`.
- Интерпретация: UTC-день без времени.
- Пустой массив означает, что дней переучета для конкурента нет.

## Поведение API

- `POST /api/konks` принимает `recountDays` при создании конкурента.
- `PATCH /api/konks/id/:id` принимает `recountDays` для изменения календаря.
- Удаление всех дней переучета выполняется тем же `PATCH`, передавая `recountDays: []`.
- Во всех ответах, где возвращается `Konk`, поле `recountDays` присутствует в формате массива дат.

## Влияние на аналитические данные

- Для дат из `recountDays` продажи конкурента считаются `0`.
- Дни переучета не вырезаются из диапазона и участвуют в агрегатах как обычные календарные дни.
- Правило применяется к расчетам продаж конкурента в:
  - графиках и сводках SKU;
  - сводках товарных групп (`skugr`);
  - сравнении конкурент vs Btrade по продажам.

## Влияние на Excel-отчеты

- В Excel по продажам ячейки продаж конкурента за день переучета:
  - содержат `0`;
  - выделяются фиолетовой заливкой.
- Подсветка используется как явный индикатор, что значение занулено бизнес-правилом переучета, а не отсутствием данных.

## Ожидания к UI/UX

- В форме конкурента нужен редактор массива дат `recountDays`:
  - добавление даты;
  - удаление даты;
  - полная очистка списка.
- Отображение списка дат должно быть стабильным и однозначным для UTC-дня.
- При редактировании важно не смешивать `date` и `datetime`: поле всегда работает как календарная дата без времени.

## Контроль консистентности на фронте

- Не преобразовывать `recountDays` в локальные даты с часовым смещением при сохранении.
- Не отправлять дубликаты дат в одном запросе.
- При показе аналитики/таблиц использовать серверные значения продаж как источник истины; не пытаться повторно пересчитывать правило переучета на клиенте.
