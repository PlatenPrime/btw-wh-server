# Модуль Events (Аудит действий)

## Описание модуля

Модуль `events` ведёт сквозной журнал действий пользователей на складе. Запись создаётся сервером после успешной бизнес-операции (мутации и отдельные значимые чтения) и хранится в отдельной append-only коллекции. Модуль решает задачу прозрачности: кто, в каком домене и что сделал — без опоры на ops-логи (Pino) и без смешения с локальной историей сущностей вроде `Ask.events`.

## Сущности модуля

### Event (Событие)

Документ аудита. Содержит `userId` — ссылку на пользователя-актора; `userData` — неизменяемый снимок пользователя на момент события (`_id`, `fullname`, опционально `telegram` и `photo`); `department` — строковый домен/модуль (`constants`, `poses`, `asks` и т.д.), а не орг-отдел; `description` — человекочитаемое описание произошедшего. Документ имеет стандартные timestamps создания и обновления.

### EventUserData

Вложенный снимок пользователя внутри события. Нужен, чтобы журнал оставался читаемым, даже если позже у пользователя сменятся имя, telegram или фото.

## Связи между сущностями

- **Event → User**: через `userId` (ObjectId ref). Связь односторонняя: User не хранит список своих событий.
- **Event.userData**: денормализованный снимок, построенный из User в момент записи. Не обновляется при изменении User.
- Глобальные Events не заменяют embedded-историю заявок (`Ask.events`) и другие локальные журналы сущностей.

## Концепции и принятые решения

### Актор только из JWT

Идентификация того, кто выполнил действие, берётся из `req.user.id` после `checkAuth` (JWT в заголовке `Authorization: Bearer <token>`). Клиент не передаёт и не может подставить `userId` актора в body/query для записи аудита. Дополнительный user id в запросах для аудита не требуется.

### Запись только внутри сервера

Публичного `POST /api/events` нет: клиент не может писать в журнал напрямую. Создание выполняется внутренним util (`createEventUtil`) из контроллеров после успешной операции.

### Fail-soft

Ошибка записи события логируется и не должна ронять успешную бизнес-операцию. Пустые `department`/`description`, невалидный или отсутствующий user — событие пропускается.

### Department как домен модуля

`department` — строка имени домена/HTTP-модуля, к которому относится действие. Отдельной сущности «отдел организации» в системе нет.

### Append-only

В первой версии нет публичных update/delete событий. Журнал предназначен для чтения (список с фильтрами и получение по id).

### Покрытие мутаций

После успешной аутентифицированной мутации (POST, PUT, PATCH, DELETE) соответствующий контроллер вызывает `createEventUtil` с `department`, равным имени модуля (`poses`, `asks`, `pallets` и т.д.). Описание (`description`) формулируется на украинском языке и включает конкретные данные операции: идентификаторы, артикулы, названия, количества, агрегаты для массовых действий.

Не пишутся события для: публичных `POST /api/auth/login` и `POST /api/auth/register` (нет JWT-актора), неаутентифицированных `GET /api/browser/*`, обычных GET-чтений и отчётов, а также для чтения самого журнала `/api/events`.

Для массовых и фоновых операций создаётся одно событие на HTTP-запрос (или на фактическое завершение фоновой задачи, если результат известен только после неё), с итоговыми счётчиками в описании.

### Язык описаний

Поле `description` всегда на украинском: глагол действия, тип сущности и ключевые поля из результата или входных данных. Пароли и токены в описание не попадают.
