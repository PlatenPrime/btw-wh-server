# Модуль SKU Slices (Срезы SKU по датам)

## Назначение

Модуль хранит ежедневные срезы остатков и цен по товарам конкурентов, представленным сущностью SKU. Один документ среза соответствует паре «конкурент (`konkName`) + календарная дата» и содержит объект `data`, в котором ключи — канонические идентификаторы товаров (`Sku.productId`, формат `{konkName в нижнем регистре}-{сырой id с сайта конкурента}`), значения — остаток и цена, полученные тем же способом скрапинга страницы товара, что используется для аналогов (по URL SKU).

Excel-выгрузки и большинство JSON-эндпоинтов используют только срезы конкурента; данные Btrade в них не участвуют. **Исключение по Btrade:** эндпоинты `konk-prod/*-chart-data` сравнивают агрегат конкурента и Btrade по производителю: конкурент — сумма по SKU с **`konkName` и `prodName`, точно совпадающими** с query после trim (регистр и полное совпадение строк важны); Btrade — сумма по **всем артикулам из Art**, у которых `prodName` после trim совпадает с query `prod` **без учёта регистра**, данные из `BtradeSlice`. Связь SKU↔артикул через `btradeAnalog` там **не используется**; смысл и форма рядов согласованы с analog `konk-btrade` comparison. Excel по query `konk`/`prod` строятся **только** из срезов конкурента, но отбор SKU для них — с тем же **точным** совпадением `konkName`/`prodName` с параметрами. Для срезовых Excel — метрики `Залишок` и `Ціна`, для sales Excel — `Продажі`, `Ціна`, `Виручка`.

## Сущности

### SkuSlice

Документ дневного среза по одному конкуренту. Поля: `konkName`, `date` (начало суток по UTC), `data` — вложенный объект с ключами `productId` и значениями `{ stock, price }`. Уникальный составной индекс по `(konkName, date)`.

## Связи

- **Sku:** каждая запись в `data` соответствует одному SKU с тем же `productId` и тем же `konkName`, что и документ среза. Доступ к точке среза по конкретному товару на фронте обычно идёт через `skuId`: сервер загружает SKU, читает `productId` и `konkName`, затем ищет значение в документе среза. Поле `btradeAnalog` используется в других сценариях данных, не в `konk-prod/*-chart-data`.

- **Art:** для `konk-prod/*-chart-data` множество артикулов Btrade — все документы Art с `prodName`, совпадающим с query `prod` после trim **без учёта регистра**; отбор SKU конкурента по тому же `prod` в query — **с точным** `Sku.prodName`.

- **Парсеры групп конкурентов:** при заполнении SKU из обхода страниц группы в документ записывается `productId`, согласованный с правилом канонизации (склейка нормализованного имени конкурента и id товара со страницы). Это обеспечивает стабильные ключи в `data` без кодирования URL.

- **Skugr (товарная группа):** состав группы задаётся массивом `skus` в документе Skugr; отчёты и агрегаты по группе в этом модуле используют только этот список и не требуют передавать `konk`/`prod` в query (они хранятся на группе и в каждом SKU).

## Сбор данных

По расписанию (cron) для каждого `konkName`, присутствующего у SKU в базе, формируется задача среза; конкуренты из конфига **исключений** для сценария `skuSlices` (с нормализацией имени, как в коде крона) в обход **не попадают**. Список конкурентов для запуска — по сути `distinct konkName` по SKU после этого фильтра; сами задачи по конкурентам выполняются **параллельно**.

Скрапингом обходятся **не все** SKU конкурента, а только те, чьи `_id` входят в объединение массивов `skus` документов **Skugr** с тем же `konkName` и флагом **`isSliced: true`**. Если для конкурента нет ни одного такого SKU, документ среза на ключ даты всё равно может быть создан/обновлён, но поле `data` не наполняется позициями.

Ключ `date` в документе среза для крона — **следующий** календарный день в `Europe/Kiev` относительно момента запуска (`toNextKyivSliceDate`), а не календарная «сегодня» без этого сдвига.

Внутри одного конкурента отобранные SKU обходятся **последовательно** с паузой **500–1500** мс (случайный jitter) между HTTP-запросами к сайту конкурента. Неподдерживаемый для скрапинга конкурент приводит к прерыванию обработки этого конкурента после первой ошибки типа «неподдерживаемый konk»; прочие ошибки по отдельным SKU логируются, остальные позиции продолжают обрабатываться.

Часовой пояс cron: **Europe/Kiev**, расписание — **ежедневно в 20:00** по этому поясу.

## HTTP и отчёты

Чтение срезов для API и отчётов в ряде сценариев идёт через aggregation-пайплайны MongoDB: либо считается и нарезается страница по ключам `data` (`GET /api/sku-slices`), либо в приложение передаётся только подмножество `data` — одна позиция по `productId`, список позиций по группе SKU / Skugr и т.п. — без полной передачи всего объекта `data` каждого затронутого документа среза. Общие стадии проекции вынесены в утилиту модуля `sliceDataAggregationStages`. Для Btrade при chart-data используется проекция среза по списку артикулов в модуле `btrade-slices` (сужение поля `data` в пайплайне). Дневные ряды конкурента по произвольному списку SKU (без Btrade) вынесены в общий хелпер, переиспользуемый отчётом Skugr `daily-summary`.

Предусмотрены только операции чтения: JSON (срез по конкуренту и дате, ряды по одному SKU, продажи и выручка по тем же правилам, что и для срезов аналогов, **дневные агрегаты по товарной группе Skugr** для графиков, **два эндпоинта chart-data по konk+prod** с рядами конкурент vs Btrade) и **шесть** Excel-выгрузок: по одному SKU (остатки и продажи), по группе SKU с фильтром конкурент + производитель (остатки и продажи), по **списку `skugr.skus`** (остатки и продажи). Заголовки колонок метаданных в Excel на украинском языке и отражают поля SKU (назва, посилання, ідентифікатор, конкурент, виробник).

Для групповой выгрузки по `konk` + `prod` (`/api/sku-slices/konk/excel`) после всех блоков SKU добавляется строка `Підсумок` с агрегатами по колонкам `Різниця` и `Різниця, %`. Для выгрузки по **Skugr** (`/api/sku-slices/skugr/.../slice-excel`) итоговая строка устроена так же.

В Excel по остаткам (`slice-excel` для одного SKU, для `konk`/`prod` и для Skugr) в строке метрики **Залишок** для ячеек по датам действуют заливки: светло-красная — если по сравнению с предыдущим днём выбранного периода остаток вырос (интерпретация «день поставки»); светло-зелёная — если календарная дата колонки совпадает с днём создания записи SKU в смысле того же календарного дня, что и у срезов (Europe/Kiev, как у `toSliceDate`). Если оба условия выполняются для одной ячейки, применяется только зелёная заливка (новинка важнее). Строка **Ціна**: по сравнению с предыдущим днём периода снижение цены — зелёный шрифт, повышение — красный; заливки строки **Залишок** на ячейки цены не распространяются.

## Роли

Эндпоинты модуля ориентированы на чтение с аутентификацией и ролью USER, аналогично JSON-роутам модуля срезов аналогов.

## API эндпоинты

- **GET `/api/sku-slices`** — срез по `konkName` и `date` с пагинацией (`page`, `limit`); опционально фильтр `isInvalid=true` — только позиции, триггерящие компенсирующие срезы; в ответе массив `items` (строки среза + маппинг на документы Sku по `productId`), см. [API документацию](../api/sku-slices.md).
- **GET `/api/sku-slices/sku/:skuId`** — точка среза на дату (query `date`).
- **GET `/api/sku-slices/sku/:skuId/range`** — срез за период (`dateFrom`, `dateTo`).
- **GET `/api/sku-slices/sku/:skuId/sales-by-date`** и **sales-range** — продажи и выручка на дату и за период.
- **GET `/api/sku-slices/sku/:skuId/slice-excel`** — Excel по одному SKU за период.
- **GET `/api/sku-slices/konk/excel`** — Excel по группе SKU (`konk`, `prod`, период).
- **GET `/api/sku-slices/sku/:skuId/sales-excel`** — Excel продаж по одному SKU за период.
- **GET `/api/sku-slices/konk/sales-excel`** — Excel продаж по SKU группы `konk` + `prod` за период.
- **GET `/api/sku-slices/konk-prod/stock-chart-data`** и **`/konk-prod/sales-chart-data`** — JSON для графиков: агрегат SKU группы vs Btrade за период (query `konk`, `prod`, `dateFrom`, `dateTo`).
- **GET `/api/sku-slices/skugr/:skugrId/daily-summary`** — дневные суммарные остатки, продажи (шт.) и выручка по группе (`skugr.skus`, период `dateFrom`/`dateTo`).
- **GET `/api/sku-slices/skugr/:skugrId/slice-excel`** — Excel остатков по SKU из группы за период.
- **GET `/api/sku-slices/skugr/:skugrId/sales-excel`** — Excel продаж по SKU из группы за период.

Подробные форматы запросов и ответов, а также ограничения query (`dateFrom` ≤ `dateTo` на периодах, валидный ObjectId для `skuId`/`skugrId`, trim и непустота `konk`/`prod` после trim) — в [API документации](../api/sku-slices.md).
